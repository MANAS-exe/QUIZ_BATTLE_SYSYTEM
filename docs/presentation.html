"Here's the full architecture. 5 Go services, each running independently. The Flutter app talks to 3 of them via gRPC streaming and 1 via HTTP. The 5th is a background worker that sends push notifications. All inter-service communication goes through RabbitMQ — no direct service-to-service calls."

Why This Tech Stack?

Why 5 Separate Services?

MatchmakingMust stay up even during games
Quiz EngineLong-lived streaming; isolated per room
ScoringHigh-frequency writes; atomic Redis ops
PaymentPCI-adjacent; HTTP for Razorpay webhooks
NotificationFire-and-forget; no client API

Why These Technologies?

gRPCNative streaming for real-time game events
RedisSub-ms latency for leaderboard + matchmaking pool
RabbitMQTopic exchange fan-out + DLQ for failed messages
MongoDBFlexible schema; $sample for questions
Lua scriptsAtomic Redis ops (score + rank in 1 call)

STRESS: "Each service exists for a specific reason. Matchmaking handles identity and must never go down during a game. Quiz Engine manages long-lived gRPC streams — one per player. Scoring has the highest write frequency — every answer triggers a Redis update. Payment is HTTP because Razorpay sends webhooks, not gRPC. And the notification worker is pure background — no API at all."

Race Condition Prevention

The hardest engineering problems in this project:

1. Atomic Leaderboard (Lua Script)

redis.call('ZINCRBY', key, points, member)
redis.call('EXPIRE', key, 1800)
local rank = redis.call('ZREVRANK', key, member)
return rank

3 ops in 1 round-trip. No stale rank between ZINCRBY and ZREVRANK.

2. Double Round Completion (SETNX)

SETNX room:{id}:round:{n}:closed "1" EX 1800

Timer fires + last answer arrives simultaneously. Only the first goroutine proceeds.

3. Distributed Lock (UUID + Lua)

-- Acquire: SETNX with UUID owner
-- Release: Lua compare-and-delete
if redis.call('GET', key) == owner then
  return redis.call('DEL', key)
end

Prevents blind delete if lock TTL expires before release.

4. Idempotent Answers (HEXISTS)

HSETNX room:{id}:submitted:{round} {userId} {answer}
-- Scoring consumer: HEXISTS before updating leaderboard

Double-tap or network retry = same score. Never double-counted.

STRESS THESE HARD: "These are the problems that break real-time games. [Point to Lua script] This runs INSIDE Redis — 3 operations atomically. Without this, two players answering simultaneously would get stale ranks. [Point to SETNX] When 30-second timer and last answer arrive at the exact same millisecond — only one triggers round completion. [Point to lock] The UUID prevents a slow goroutine from deleting another's lock."

MongoDB — Data in Compass

7 collections with comprehensive indexes:

Show in Compass:

  1. users — rating, matches_played, matches_won, referral_code, streak, coins
  2. match_history — finalScore, rank, answersCorrect, avgResponseMs, duration
  3. questions — 150 questions, 3 difficulty levels, 12 topics
  4. payments — order_id, razorpay_payment_id, status, plan
  5. subscriptions — user_id, plan, expires_at, status
  6. device_tokens — FCM tokens per user
  7. referrals — referrer_id, referee_id, code_used

Key Points to Stress:

  • All fields populated — no 0s or nulls where real values should be
  • Unique indexes — username, referral_code (sparse), device_tokens.user_id
  • Stats persist — matches_played, matches_won survive app reinstall
  • match_history complete — winner, duration, avgResponseMs all computed
  • Referral anti-abuse — referee_id unique (can't apply twice)

  • One-tap login with Google OAuth 2.0
  • Server-side token verification via Google tokeninfo API
  • Account linking by email (password → Google merge)
  • Profile picture CDN URL stored + displayed everywhere

Daily Rewards & Streak

  • 30-day reward cycle with escalating coins + bonus games
  • Day 7: Week Warrior badge + 250 coins
  • Day 30: Monthly Master + 7-day premium trial
  • Animated claim popup on home screen
  • 30-day login calendar on Profile

Premium + Razorpay

  • Monthly ₹499 / Yearly ₹3,999
  • HMAC-SHA256 webhook signature verification
  • Idempotent webhook processing
  • Server-side quota enforcement (5 games/day free)
  • Coupon discount via referral codes

Referral System

  • 6-char codes (crypto/rand, 887M combinations)
  • 5 anti-abuse rules (self-referral, time window, cap, etc.)
  • Referrer: +200 coins, +2 bonus games
  • Referee: +100 coins, +1 bonus game
  • Pending → Claim flow (server source of truth)

FCM Push Notifications

  • 5 notification types (streak warning, daily reward, premium expiry, referral conversion, tournament)
  • Cron scheduler (7pm IST streak, 8am daily reward)
  • Referral conversion: exactly-once via atomic MongoDB flag
  • Background + foreground handling in Flutter

Server-Side Enforcement

  • Daily quota checked in JoinMatchmaking (not just UI)
  • Leaderboard capped to top 3 for free users
  • All stats persisted to MongoDB (20 fields)
  • Late joiner gets current question + timer on connect

And critically — server-side enforcement everywhere. Quota is checked on the backend, not just hidden in the UI."

Hardest Bugs & How I Fixed Them

35 bugs documented. Here are the ones worth discussing:

#BugRoot CauseFix
2 Leaderboard all 0s after microservice split Quiz handler and scoring consumer both used HSETNX on same Redis key Separated into submitted (instant) and answers (scoring) keys
7 3+ players in same room Concurrent ZPOPMIN without locking Distributed lock (SETNX + UUID + Lua release)
17 Questions repeating in same match MongoDB $sample re-visits docs at >5% pool Fisher-Yates shuffle in Go, cross-bucket exclusion
19 Razorpay crash on Android Callbacks fire during Activity onResume before Flutter tree ready Future.delayed(Duration.zero, ...) defers to next tick
28 MatchFound never delivered to players JoinMatchmaking deleted the SubscribeToMatch channel Removed the delete; SubscribeToMatch overwrites on reconnect
32 Stats reset on app reinstall Only SharedPreferences, no MongoDB sync Added GET/POST /user/stats; push after every save, pull on login

"I documented 35 bugs throughout development. Let me highlight the hardest ones. [Point to #2] After splitting into microservices, all scores showed 0 — because both services were writing to the same Redis key. [Point to #7] Matchmaking race condition — two goroutines popping the same players. Fixed with a distributed lock using UUID ownership. [Point to #17] MongoDB's $sample was repeating questions — replaced with Fisher-Yates shuffle. [Point to #28] This was the most recent — players matched but never got the event. JoinMatchmaking was accidentally deleting the subscription channel."

Pre-Demo Checklist

Run before presenting:

# 1. Start everything
docker compose up --build -d

# 2. Verify all healthy
docker compose ps

# 3. Open these tabs BEFORE the demo:
#    - MongoDB Compass → localhost:27017 → quizdb
#    - RabbitMQ UI → http://localhost:15672 (guest/guest)
#    - Two Android emulators running
#    - Redis CLI: docker compose exec redis redis-cli

# 4. Reset quotas (so demo users can play)
docker compose exec mongodb mongosh quizdb --eval "db.users.updateMany({}, {\$set: {daily_quiz_used: 0, last_quiz_date: ''}})"

# 5. Flush stale matchmaking pool
docker compose exec redis redis-cli DEL matchmaking:pool

# 6. Start Flutter on both emulators
cd flutter-app && flutter run -d emulator-5554  # Terminal 1
cd flutter-app && flutter run -d emulator-5556  # Terminal 2

"Before the demo, make sure all services are up, Compass and RabbitMQ UI are open in browser tabs, both emulators are running, and quotas are reset. This avoids any 'daily limit reached' surprises during the live demo."
