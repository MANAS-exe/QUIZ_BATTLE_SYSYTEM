Technology 01

Go (Golang)

Language Backend

What Is Go?

Go is a statically typed, compiled programming language designed at Google by Robert Griesemer, Rob Pike, and Ken Thompson. Publicly announced in 2009, version 1.0 arrived in 2012. Go was created to solve problems Google faced with large-scale development: slow C++ compilation, complex Java dependency management, and the difficulty of writing safe concurrent programs.

Go compiles directly to machine code — a single static binary with no external dependencies. No virtual machine, no interpreter, no runtime library to install. You compile, copy the binary to any compatible machine, and it runs.

Formal Definition Go is an imperative, concurrent, garbage-collected programming language with strong static typing, structural subtyping (interfaces), and first-class functions. It targets systems programming with an emphasis on simplicity, readability, and efficient concurrency through goroutines and channels.

Key Concepts

1. Static Typing & Compilation

In a statically typed language, every variable has a type known at compile time. If you declare var x int = 5, the compiler knows x is an integer. Trying x = "hello" produces a compile-time error — the bug is caught before the program ever runs. Compilation means translating source code into machine code once (at build time), so the CPU can execute it directly.

Analogy Compiled (Go, C) = Translating a full book from Hindi to English once, then distributing the English version. Anyone reads it instantly.
Interpreted (Python, JS) = Hiring a live translator who reads each Hindi sentence aloud and translates on the spot. Faster to start, slower overall, and you need the translator every time.

2. Goroutines — Lightweight Concurrency

A goroutine is a lightweight thread of execution managed by the Go runtime, not the operating system. You start one by writing go functionName(). The Go runtime multiplexes thousands of goroutines onto a small number of OS threads using an M:N scheduling model. A goroutine starts with only ~2 KB of stack space (versus ~1 MB for an OS thread). The stack grows and shrinks dynamically — you can run millions of goroutines on a single machine.

Analogy OS Threads = Hiring a full chef per order. Each needs their own workstation, knife set, and stove burner. You fit maybe 8 chefs in the kitchen.
Goroutines = 8 chefs (OS threads) but 500 order tickets (goroutines) on a rail. Each chef picks up a ticket, works on it, puts it back when waiting (water to boil), and picks up the next. The kitchen manager (Go scheduler) coordinates.

3. Channels — Communication Between Goroutines

A channel is a typed conduit for sending and receiving values between goroutines. For unbuffered channels, the sender blocks until a receiver is ready, and vice versa. This synchronized communication eliminates the need for explicit locks in many cases.

ch := make(chan int)       // unbuffered channel of ints
go func() { ch <- 42 }()  // goroutine sends (blocks until received)
value := <-ch               // main goroutine receives (blocks until sent)

4. sync.Once — Execute Exactly Once

sync.Once guarantees a function runs exactly once, regardless of how many goroutines call it concurrently. Internally: atomic flag + mutex. The first caller executes the function and sets the flag; all subsequent callers return immediately.

Usage in Quiz Battle — Current State 5 backend services, each a compiled Go binary: Matchmaking (gRPC :50051 / HTTP :8080), Quiz Engine (gRPC :50052), Scoring (gRPC :50053), Payment (HTTP :8081), Notification Worker (background consumer + cron scheduler). The goroutine-per-player model handles concurrent gRPC streams. sync.Once ensures only one game loop starts per room. HTTP handlers (matchmaking-service): /auth/google, /referral/*, /device/token, /user/stats, /user/change-password (new — JWT-authenticated password update), /tournament/*, /leaderboard. Password management: bcrypt hashing on registration; seed script uses upsert semantics so re-running it resets any test user's password to speakx123 — the canonical dev-recovery tool.

The Rejected Alternatives

Alternative: Node.js

What it is: Node.js is a JavaScript runtime built on Chrome's V8 engine. It uses an event-driven, non-blocking I/O model with a single-threaded event loop. When an I/O operation starts, Node registers a callback and moves on. When I/O completes, the callback is queued and executed when the main thread is free.

Why it wasn't chosen: Quiz Battle's game server does CPU-intensive timer management (30 timer ticks per room per round) and concurrent stream management. On a single-threaded event loop, computing scores for 10 players blocks timer ticks for all other rooms. Node's worker threads could help but add complexity equivalent to Go's goroutines without the language-level elegance. JavaScript's dynamic typing means type mismatches in gRPC messages are caught at runtime, not compile time.

When Node.js IS the right choice: REST APIs with mostly I/O-bound operations, real-time chat with socket.io, rapid prototyping, serverless functions.

Alternative: Java (Spring Boot)

What it is: Java compiles to bytecode executed by the JVM. Spring Boot is the dominant Java microservices framework. The JVM uses a Just-in-Time (JIT) compiler to convert hot code paths to native code at runtime. Java uses OS-level threads (~1 MB stack each). Modern Java 21+ introduced Virtual Threads (Project Loom) — lightweight threads similar to goroutines, but the ecosystem is still adapting.

Why it wasn't chosen: JVM startup takes 2–5 seconds (Spring Boot: 5–15 seconds); Go starts in under 100ms. Docker images are 200–400 MB (JDK + fat JAR) versus 10–20 MB for Go. Spring Boot's annotation-driven DI adds boilerplate for simple microservices.

When Java IS the right choice: Large enterprise systems with complex business logic, teams with existing Java expertise, long-running applications where JIT warm-up is amortised.

Alternative: Python (FastAPI / Django)

What it is: Python is dynamically typed and interpreted. The critical limitation is the Global Interpreter Lock (GIL) — a mutex that allows only one thread to execute Python bytecode at a time, even on a multi-core machine. Python is typically 10–100× slower than Go for CPU-bound operations.

Why it wasn't chosen: The GIL prevents true parallelism — 10 concurrent scoring operations would run sequentially. Dynamic typing means gRPC schema mismatches surface at runtime. Unacceptable latency for a real-time game server.

When Python IS the right choice: Machine learning (PyTorch, TensorFlow), scripting, rapid prototyping, I/O-dominated workloads.

Final Verdict — Why Go Go gives the best combination for a real-time game server: compiled speed, goroutines for per-player concurrency, first-class gRPC support, small Docker images (15 MB vs 300 MB for Java), and fast compilation (~3 seconds per service). 5 services, 14 HTTP endpoints, one language.

Technology 02

gRPC & Protocol Buffers

Protocol Communication

What Is RPC?

Remote Procedure Call lets a program on one computer execute a function on another as if it were local. The caller sends parameters over the network, the remote machine executes the function, and the result is returned. The key insight is location transparency: the programmer writes result = server.CalculateScore(answer) without knowing if it runs locally or across the network.

What Is gRPC?

gRPC is Google's open-source, high-performance RPC framework (released 2015, now a CNCF project). It uses HTTP/2 for transport and Protocol Buffers (Protobuf) for serialisation.

HTTP/2 — The Transport Layer

HTTP/1.1 allows only one request per TCP connection at a time (head-of-line blocking). HTTP/2 introduces multiplexing: a single TCP connection carries many requests and responses simultaneously as independent streams. This is essential for gRPC streaming — one connection carries a continuous flow of game events without blocking.

Protocol Buffers — The Serialisation Format

You define schemas in .proto files; the Protobuf compiler generates typed code in Go, Dart, Java, etc. The wire format is compact binary — fields are identified by number (not name), making it 3–10× smaller than JSON. Identical passwords produce different hashes because the salt is unique.

syntax = "proto3";

service QuizService {
  rpc SubmitAnswer(SubmitAnswerRequest) returns (SubmitAnswerResponse);
  rpc StreamGameEvents(StreamRequest) returns (stream GameEvent);
}

Usage in Quiz Battle — Current State StreamGameEvents (server-streaming) is the real-time backbone. The server pushes five event types over one persistent HTTP/2 stream: QuestionBroadcast (new question each round), TimerSync (server-authoritative time correction every second), RoundResult (correct answer + fastest player), LeaderboardUpdate (score changes), MatchEnd (final rankings + winner). Late-joiner catch-up: when a player reconnects mid-round, gameRoom.currentQuestion stores the active question event so the server can send it to any new subscriber immediately. Reconnect dedup guard: the Flutter client's _onQuestion handler checks if it is already displaying the same round and question ID before resetting state — prevents the catch-up event from restarting the timer and clearing the player's already-submitted answer.

The Rejected Alternatives

Alternative: REST (Representational State Transfer)

What it is: REST uses standard HTTP methods (GET, POST, PUT, DELETE) on resource URLs. Data is typically JSON. REST is request-response — the client asks, the server answers, the connection closes. For real-time server-push, REST requires polling (client asks repeatedly), wasting bandwidth and adding latency. No built-in schema contract — the server can change JSON structure and break clients without compile-time warnings.

When REST IS the right choice: Standard web APIs, CRUD applications, public APIs where human-readability matters, browser-based apps.

Alternative: JSON

What it is: JSON is text-based. A message with userId (10 chars), round (int), answerIdx (int), and timestamp (long) takes ~85 bytes in JSON but ~25 bytes in Protobuf. In Quiz Battle, thousands of messages flow per game session — with 10 players × 10 rounds × multiple event types, JSON overhead accumulates. More critically, JSON has no schema — type mismatches surface at runtime.

When JSON IS the right choice: Public REST APIs, config files, browser-based apps where human readability matters.

Alternative: WebSockets

What it is: WebSocket (RFC 6455) provides full-duplex communication over a single long-lived TCP connection via an HTTP Upgrade handshake. Either side can send frames at any time. However, WebSocket is a raw pipe — no concept of request/response, no method names, no status codes, no error handling. You must design your own message protocol on top. WebSocket also has no built-in multiplexing (one WebSocket = one stream), while HTTP/2 multiplexes many logical streams over one TCP connection.

When WebSockets ARE the right choice: Browser-based real-time apps (gRPC doesn't work in browsers without grpc-web proxy), chat applications, collaborative editing, live dashboards.

REST (HTTP/1.1):

  Client ──[Request]───► Server ──[Response]───► Client   (repeat for every question)

Final Verdict — Why gRPC gRPC provides a type-safe schema (Protobuf), native streaming (no custom protocol design), binary encoding (3–10× smaller than JSON), auto-generated client/server code, and one unified protocol for both request-response and streaming. The reconnect architecture (Technology 14) wraps the stream with automatic recovery — making the combination robust for a mobile game on unreliable networks.

Technology 03

Redis

Database In-Memory

What Is Redis?

Redis (Remote Dictionary Server) is an open-source, in-memory data structure store created by Salvatore Sanfilippo in 2009. It stores data entirely in RAM, making it orders of magnitude faster than disk-based databases: a typical operation completes in under 1 millisecond. Redis supports rich data structures: strings, hashes, lists, sets, sorted sets, streams, and more.

Formal Definition Redis is a networked, single-threaded (command execution), in-memory data structure server supporting persistence to disk, replication, Lua scripting, LRU eviction, transactions, pub/sub messaging, and cluster mode. It exposes a command-based protocol (RESP) over TCP.

Key Data Structures Used

Sorted Sets: Members with floating-point scores, kept in ascending order. Implemented as a skip list + hash table. Used for matchmaking pool (score = rating) and leaderboards (score = game points).

Hashes: Field-value maps under a single key. Used for player data per room and idempotency guards (HEXISTS checks if a player already answered this round).

Lists: Ordered collections (doubly-linked list). LPOP is used per round to destructively consume question IDs — guaranteeing each question appears exactly once within a match.

Strings with SETNX: Set only if key doesn't exist. Used for distributed locks and round-completion guards — prevents two goroutines from concurrently ending the same round.

Lua Scripts: Atomic multi-command execution on the Redis server. Used for leaderboard update (ZINCRBY + EXPIRE + ZREVRANK in one atomic operation).

Usage in Quiz Battle — Current State Redis holds all ephemeral game state. Every key has a 30-minute TTL for automatic cleanup. Key namespaces:
  • matchmaking:pool — sorted set of waiting players (score = rating)
  • player:{userId} — hash of username, rating, joined_at
  • room:{roomId}:* — questions list, players hash, leaderboard sorted set, round locks
  • answer:{roomId}:{round}:{userId} — idempotency guard (SETNX prevents double-scoring)
  • user:{userId}:daily_quota — remaining free games today (mirrors MongoDB for fast quota checks)
  • user:{userId}:plan — "premium" or free (fast plan gate without MongoDB round-trip)
  • matchmaking_create — distributed lock preventing two goroutines from creating rooms simultaneously
Redis is the fast scratchpad; MongoDB is the permanent filing cabinet.

The Rejected Alternatives

Alternative: Memcached

What it is: Memcached stores data as flat key-value pairs where values are opaque byte blobs. It uses a slab allocator and LRU eviction. Memcached is multi-threaded, giving it an edge in raw throughput on multi-core machines.

What Memcached CANNOT do: No sorted sets (needed for leaderboards), no hashes (needed for player data), no lists (needed for question queues), no atomic multi-step Lua scripts, no persistence, and no pub/sub. To implement a leaderboard you'd fetch all scores to the application, sort in Go, and write back — a multi-step non-atomic process with race conditions.

When Memcached IS the right choice: Simple key-value caching (session data, API responses), especially at extreme scale where multi-threaded performance matters more than data structure richness.

Alternative: In-Process Memory (Go maps / sync.Map)

The fundamental problem — shared nothing: The Matchmaking service adds players to the pool; the Quiz Engine reads from the pool. They're separate processes on separate ports (50051 and 50052). Go maps in one process are invisible to another. There's no shared memory between services. You'd need to either merge all services into one monolith or add a custom network protocol to share state — essentially rebuilding Redis.

When in-process memory IS the right choice: Single-process applications, per-request caches, data that doesn't need to be shared across services.

Final Verdict — Why Redis Sub-millisecond latency, rich data structures (sorted sets for leaderboards + matchmaking, lists for question queues, hashes for player data), Lua scripting for atomic operations, built-in TTL for cleanup, and shared access across all 5 services. Also serves as a fast cache for daily quota and plan status to avoid MongoDB round-trips on every matchmaking join.

Technology 04

RabbitMQ

Message Broker

What Is a Message Broker?

A message broker receives messages from producers, stores them in queues, and delivers them to consumers. Producers and consumers are decoupled in time (don't need to be online simultaneously), space (don't need to know each other's addresses), and synchronisation (producer doesn't block waiting for consumer).

What Is RabbitMQ?

RabbitMQ is an open-source message broker written in Erlang, implementing the AMQP 0-9-1 protocol. Core concepts: Exchange (receives messages and routes them), Queue (stores messages until consumed), Binding (rule connecting an exchange to a queue), and Consumer (receives and processes messages). Quiz Battle uses a single Topic Exchange named sx. Topic exchanges route messages based on pattern matching: notification.* matches notification.streak_warning and notification.daily_reward. Messages are acknowledged (ACK) after successful processing or negatively acknowledged (NACK) for retry, with a Dead Letter Queue after repeated failures.

The Rejected Alternatives

Alternative: Apache Kafka

What it is: Kafka stores messages in topics divided into partitions — ordered, immutable, append-only logs on disk. Consumers read at their own pace by tracking their offset. Messages are never deleted by consumption — they're retained for a configurable period. Multiple consumers can read the same messages independently.

Key architectural difference: In RabbitMQ, the broker tracks delivery (ACK-based — once ACKed, message is deleted). In Kafka, the consumer tracks its own position (offset). This enables replay but requires managing offsets and Kafka Transactions for exactly-once semantics.

Why Kafka wasn't chosen: Kafka requires ZooKeeper or KRaft for cluster coordination — an additional service to deploy. It shines at millions of messages/second with weeks of retention. Quiz Battle produces hundreds of messages/second with no retention needed. No topic routing flexibility either — you'd need separate topics or consumer-side filtering instead of RabbitMQ's wildcard bindings.

When Kafka IS the right choice: Log aggregation, event sourcing (rebuilding state from history), real-time analytics pipelines, Change Data Capture (CDC), any scenario requiring replay capability.

Alternative: Redis Pub/Sub

The critical limitation — fire and forget: If no consumer is subscribed when a message is published, the message is lost forever. There is no queue to buffer it. If the scoring service restarts and misses an answer.submitted message, that answer is never scored — silently lost. No ACK/NACK, no retry, no dead letter queue, and no way to ensure exactly-once processing.

When Redis Pub/Sub IS the right choice: Real-time notifications where loss is acceptable (sports scores, stock tickers), cache invalidation broadcasts, inter-process signalling within a single application.

Alternative: Direct gRPC Calls Between Services

Problems with direct calls: (1) Tight coupling — Quiz Engine must know Scoring Service's address. (2) Synchronous blocking — Quiz Engine waits for scoring before proceeding; 10 simultaneous answers × 50ms each = 500ms stall. (3) No fan-out — match.finished needs to reach Scoring AND Notification Worker; direct calls require the Quiz Engine to call both sequentially. (4) No retry — if Scoring is down, the direct call fails and the answer is lost. (5) Cascade failures — Scoring overload back-pressures to Quiz Engine, freezing all players' timers. All five problems are solved by a message broker.

When direct calls ARE the right choice: Synchronous operations where you need the result immediately (auth checks before proceeding), simple two-service systems with no fan-out, operations where broker latency is unacceptable.

Final Verdict — Why RabbitMQ Per-message ACK/NACK (critical for exactly-once scoring), topic exchange with wildcard routing (flexible fan-out to multiple consumers), dead letter queues (failure isolation), durable message storage (survives restarts), and a management UI (observability during demos). 7 event types, 5 consumers, one broker.

MongoDB

Database

What Is MongoDB?

MongoDB is a document-oriented NoSQL database that stores data as flexible, JSON-like documents (BSON — Binary JSON). Each document can have a different structure; there's no rigid schema. Documents are grouped into collections. MongoDB supports dynamic schemas, rich queries, secondary indexes, aggregation pipelines, and change streams.

The Rejected Alternatives

Alternative: PostgreSQL

What it is: PostgreSQL stores data in rows and columns with a fixed schema. It supports full ACID transactions, complex SQL queries with JOINs, foreign keys, and constraints. Uses Write-Ahead Logging (WAL) for durability, MVCC for concurrency, and 8 KB page-based storage on disk.

Schema rigidity: Every row in a table has exactly the same columns. Phase 2 added 8 new fields to the user document. In PostgreSQL, that's 8 ALTER TABLE statements with potential data backfills. In MongoDB, you just start writing documents with the new fields — old documents without them continue to work fine.

When PostgreSQL IS the right choice: Financial systems requiring strict ACID guarantees, complex relationship queries (many-to-many), reporting-heavy systems, and mature projects with stable schemas.

Alternative: MySQL

What it is: MySQL is the world's most popular open-source relational database. Its InnoDB engine supports ACID transactions and MVCC. MySQL historically prioritised simplicity and speed for read-heavy workloads. PostgreSQL generally has richer features (better JSON support, more index types, materialized views).

Why not chosen: Same schema rigidity issues as PostgreSQL. If a relational database were needed, PostgreSQL would be preferred for its richer feature set.

Final Verdict — Why MongoDB Flexible schema enabled rapid Phase 1→Phase 2 iteration across 8 collections without migration scripts. Documents map naturally to the project's data structures. Trade-off: no multi-document ACID transactions (though MongoDB 4.0+ supports them), no JOINs, less mature tooling for complex analytical queries.

Flutter & Dart

Mobile Language

What Is Flutter?

Flutter is Google's UI toolkit for building natively compiled applications for mobile (iOS, Android), web, and desktop from a single codebase. Instead of using platform-native UI components, Flutter draws every pixel itself using the Skia/Impeller rendering engine. Flutter uses Dart, supports AOT compilation for production and JIT for development (hot reload in under 1 second).

Usage in Quiz Battle — Current State Single codebase for iOS and Android. 11 screens:
  • Login — username/password or Google Sign-In
  • Home — daily reward popup (30-day calendar), streak display, quick-join, coin balance
  • Matchmaking — live lobby with waiting player list, cancel button
  • Quiz — active round: question card, 4 options, countdown timer ring, leaderboard, reconnect banner
  • Leaderboard — between-rounds score display with correct answer reveal
  • Results — final scores, winner banner, "No correct answers" state when all players answer wrong
  • Spectating — forfeited-player view waiting for match to end
  • Profile — 5 tabs (PROFILE, LAST MATCH, BADGES, STREAK, REFERRAL) + change-password dialog (lock icon in AppBar) + match history: premium users see last 3 matches, free users see last 1
  • Premium — Razorpay checkout, subscription status, feature comparison
  • Global Leaderboard — top players by rating
  • Tournament — list of upcoming/active tournaments, join flow, entry fee display
Service layer: 3 gRPC stubs (Auth, Matchmaking, Quiz), 1 HTTP client (REST endpoints), ReconnectService (exponential backoff — see Technology 14), GoRouter for navigation with auth guards.

The Rejected Alternatives

Alternative: React Native

What it is: React Native renders actual native platform components. Originally used a bridge (JS thread → JSON serialisation → native thread). The new architecture (Fabric + JSI) replaces the bridge with direct C++ calls, reducing UI update latency. Still in progressive rollout.

Why React Native wasn't chosen: (1) Animation performance — countdown timer ring, leaderboard reordering, and reward popups require smooth 60fps; Flutter's direct Skia rendering has no JS–native communication overhead. (2) gRPC support is less mature in React Native — requires native module bridges. (3) Dart's AOT compilation catches more issues at build time than TypeScript.

When React Native IS the right choice: Teams with JavaScript/React expertise, apps needing exact native platform look-and-feel, complex native module integrations (Bluetooth, camera), web+mobile code sharing.

Alternative: Native Development (Kotlin for Android + Swift for iOS)

What it is: Separate apps per platform using official languages and tools. Best performance, full platform API access, and the best debugging tools (Android Studio Profiler, Xcode Instruments).

Why it wasn't chosen: Two codebases to maintain. All game logic (gRPC clients, state management, timer synchronisation, reconnection, daily reward calculations, referral system) would need to be implemented and tested twice. For a project with a deadline, this doubles effort without proportional benefit.

When native IS the right choice: Peak-performance apps (3D games, video editors), deep OS integration (widgets, watch complications), platform-specific UX requirements, large teams with separate iOS/Android expertise.

Final Verdict — Why Flutter One codebase for both platforms halving development effort. Hot reload for rapid iteration. Mature gRPC and Protobuf packages. 11 screens, 5 service connections, zero code duplication between iOS and Android.

Technology 07

Riverpod

State Management

What Is Riverpod?

Riverpod is a reactive caching and dependency injection framework for Flutter/Dart. It uses providers — globally declared, lazily evaluated, automatically disposed units of state. A provider holds a value and notifies listeners when it changes. Widgets "watch" providers and automatically rebuild. Riverpod is compile-safe: provider access errors are caught at build time, not runtime.

Usage in Quiz Battle — Current State AuthNotifier (StateNotifierProvider<AuthState>): Manages login/logout, userId, token, username, rating, coins, loginStreak, lastLoginDate, isPremium, premiumExpiresAt, matchesPlayed, matchesWon, dailyQuizUsed, bonusGamesRemaining, referralCode, referredByCode, matchHistory (list of up to 3 LastMatchData — premium users see all 3, free users see only 1 in the UI). Key computed getter: isEffectivelyPremium returns true if the server-confirmed premium subscription is active. Methods: login, register, googleSignIn, logout, changePassword, recordMatchResult, syncStatsFromServer, claimDailyReward, syncReferralFromServer. Persists state to SharedPreferences (v2 JSON format for match history).

GameNotifier (StateNotifierProvider<GameState>): Manages active match: phase (idle → matchmaking → starting → inRound → betweenRounds → spectating → finished), currentRound, currentQuestion, selectedAnswerIndex, remainingSeconds, roundExpired, leaderboard, lastRoundResult, matchEnd, currentAnswerStreak, maxAnswerStreak, currentWinStreak, maxWinStreak. The _onQuestion handler has a dedup guard — silently skips the event if already displaying the same round+questionId.

ReconnectStateProvider: Drives the reconnect banner (status, attempt number, countdown). Separate from GameNotifier so the banner rebuilds independently without re-rendering the whole quiz screen.

The Rejected Alternatives

Alternative: BLoC (Business Logic Component)

What it is: BLoC enforces strict unidirectional data flow. UI sends Events to the Bloc; the Bloc processes them and emits new States; the UI rebuilds. For every feature: an Events sealed class, a States sealed class, and a Bloc class. The UI uses BlocBuilder for rebuilding.

Advantages: Extremely testable, forces separation of UI and logic, great for large teams, built-in stream transformations (debounce, throttle).

Why not chosen: Significant boilerplate — a simple "submit answer" feature requires 4 files and ~80 lines before any business logic. Riverpod's StateNotifier achieves the same with one class and ~20 lines. For Quiz Battle's 2 major state holders, BLoC would triple the class count.

Alternative: Provider (Legacy Flutter Package)

What it is: Built on Flutter's InheritedWidget. Providers live in the widget tree — accessing a provider that isn't an ancestor throws a runtime ProviderNotFoundException. This error surfaces during widget tree refactors and is only caught when the code runs, not at compile time. Riverpod providers are global (no widget tree position dependency), making this error structurally impossible.

Alternative: GetX

What it is: GetX combines state management, dependency injection, and route management. Variables become reactive with .obs; listeners use Obx(() => ...). Minimal boilerplate but "magic" implicit DI and global singletons make code harder to test and reason about at scale.

Technology 09

JWT (JSON Web Tokens)

Authentication

What Is JWT?

A JWT (RFC 7519) is a compact, self-contained token with three Base64URL-encoded parts: Header (algorithm), Payload (claims like userId, expiration), and Signature (HMAC-SHA256 of header+payload using a secret key). JWTs are stateless: the server doesn't store sessions — all information needed for verification is in the token itself.

Usage in Quiz Battle — Current State After login (username/password or Google Sign-In), the server issues a JWT containing userId and exp. The Flutter app includes it as Authorization: Bearer <token> in every gRPC call's metadata and every REST HTTP call. Server-side enforcement: every gRPC handler extracts userId from the JWT context via middleware.UserIDFromContext(ctx) — never from the request body (prevents impersonation). New — change-password endpoint: POST /user/change-password requires a valid JWT; the handler calls auth.ValidateToken(token) to extract userId, then updates password_hash in MongoDB. No old password required — possession of a valid JWT is proof of identity. After a successful password change, the new password is also saved to SharedPreferences so auto-login at next launch still works.

The Rejected Alternatives

Alternative: Server-Side Sessions with Cookies

What it is: The server creates a session record stored in memory or Redis. The client receives an opaque session ID in an HTTP cookie. On every request, the browser automatically sends the cookie. The server looks up the session ID to find user data.

Why sessions weren't chosen: (1) Stateful — all 5 services would need access to a shared session store. JWTs are self-contained: any service verifies them independently. (2) Cookies don't work with gRPC — gRPC uses HTTP/2 metadata headers, not browser cookies. (3) Mobile clients — cookies are a browser concept; mobile apps use token-based auth more naturally.

JWT's trade-off — no instant revocation: With sessions, the server can instantly invalidate a user (delete the session). With JWTs, the token is valid until expiry — you can't revoke without a server-side blacklist (defeating the "stateless" benefit). Acceptable for a quiz game where tokens expire in a few hours.

Alternative: OAuth 2.0 Opaque Tokens

What it is: Opaque tokens are random strings with no embedded data. To validate one, the resource server must call the authorization server's introspection endpoint — a network request on every API call. JWTs are self-verifiable with a local signature check. No network call needed.

Technology 10

OAuth 2.0 & Google Sign-In

Identity

What Is OAuth 2.0?

OAuth 2.0 (RFC 6749) is an authorization framework that lets a third-party app access a user's resources on another service without sharing passwords. OpenID Connect (OIDC) adds authentication on top: an ID token (JWT from Google) contains verified identity claims. Google Sign-In uses OIDC.

Usage in Quiz Battle Flutter calls GoogleSignIn.signIn() → receives Google ID token → sends to POST /auth/google → backend verifies with Google's tokeninfo API → upserts user in MongoDB (creates on first login, updates on return) → issues app JWT. Google-only accounts have an empty password_hash field — they are excluded from the change-password flow.

The Rejected Alternatives

Alternative: SAML (Security Assertion Markup Language)

What it is: SAML is an older (2005) XML-based standard for exchanging authentication data between an identity provider (IdP) and a service provider (SP). It was the dominant enterprise SSO standard before OAuth 2.0/OIDC. Uses verbose XML, is designed for browser-based SSO (not mobile apps), and is much more complex to implement. Google's mobile SDKs use OIDC natively — SAML is not applicable.

When SAML IS the right choice: Enterprise environments (Okta, Active Directory), legacy systems that predate OAuth 2.0, corporate SSO between web applications.

Alternative: Custom Username/Password Only (No Google)

Why both were implemented: Google Sign-In reduces friction (one tap, no password to remember, verified email, profile picture — reduces drop-off in user onboarding). Custom auth provides a fallback for users who prefer not to link Google accounts. The POST /user/change-password endpoint serves custom-auth users only.

Razorpay

Payment Gateway

What Is Razorpay?

Razorpay is an Indian payment gateway (founded 2014) supporting UPI, credit/debit cards, net banking, wallets, and EMI. Test mode with rzp_test_* keys simulates payments without real money. Integration model: backend creates an order via API → Flutter opens checkout SDK → Razorpay processes payment → webhook notifies backend → backend verifies HMAC-SHA256 signature.

Usage in Quiz Battle Payment Service (port 8081, HTTP) creates Razorpay orders, verifies webhook HMAC-SHA256 signatures, creates/updates subscription documents in MongoDB (status: "active", expires_at), validates referral coupon codes (applies discount), and publishes payment.success to RabbitMQ. Webhook handler is idempotent — checks razorpay_payment_id existence before processing to prevent double-crediting. After payment, user:{userId}:plan in Redis is updated to "premium" so subsequent matchmaking quota checks don't require a MongoDB round-trip.

The Rejected Alternatives

Alternative: Stripe

What it is: Stripe (founded 2010) is widely considered the gold standard for developer experience globally. Uses PaymentIntents, Stripe Elements for UI, and webhook signatures (HMAC-SHA256, same as Razorpay).

Why not chosen: Stripe does not support UPI — the dominant payment method in India with over 10 billion transactions/month. The app targets an Indian audience and the project spec required demonstrating Indian payment methods. Stripe also requires a US/EU bank account for payouts.

When Stripe IS the right choice: Global applications, US/EU-focused businesses, projects where developer experience is the top priority, Stripe Connect for marketplaces.

Alternative: PayU (India)

What it is: PayU is another Indian payment gateway supporting UPI, cards, net banking, and wallets. Owned by Prosus.

Why not chosen: Razorpay has a more developer-friendly experience: cleaner API documentation, better Flutter SDK, simpler test mode setup, and a simpler webhook verification model.

Alternative: Direct Bank Integration (UPI via NPCI APIs)

Why not chosen: Direct UPI integration requires PSP registration with NPCI (months-long regulatory process), RBI guideline compliance, and direct banking partnerships. Payment gateways like Razorpay abstract all of this. For a student project, this is a non-starter.

Firebase Cloud Messaging (FCM)

Push Notifications

What Is FCM?

Firebase Cloud Messaging is Google's free service for sending push notifications to Android, iOS, and web. Your server sends a message to FCM's servers via Firebase Admin SDK, and FCM delivers it to the device. FCM handles maintaining persistent connections to billions of devices — you don't manage any device connections yourself.

Usage in Quiz Battle The Notification Worker service (a background Go binary with no HTTP port) consumes from RabbitMQ and dispatches via Firebase Admin SDK. 5 notification types:
  • Streak warning — cron job fires nightly; finds users with active streaks; sends "Your streak expires tonight!"
  • Daily reward ready — cron job fires daily at 9 AM; sends "Your reward is waiting!"
  • Premium expiry — cron job checks subscriptions expiring in 24 hours; sends "Your premium expires tomorrow"
  • Referral conversion — triggered by match.finished event; when a referred user completes their first match, both referee and referrer get a notification
  • Tournament reminder — on-demand when a user joins a tournament starting within 1 hour
Flutter registers the FCM device token at login via POST /device/token. The Notification Worker uses multicast for bulk sends (up to 500 tokens per FCM batch limit). iOS notifications are routed through APNs automatically by FCM.

The Rejected Alternatives

Alternative: Apple Push Notification Service (APNs) — Direct

What it is: Every iOS push notification ultimately goes through APNs — even FCM routes iOS notifications through APNs. Direct APNs integration requires a persistent HTTP/2 connection and JWT-based authentication with your Apple Developer key. APNs only covers Apple devices. Using FCM alone covers both platforms — FCM routes iOS notifications through APNs automatically, eliminating the need to manage two separate push systems.

Alternative: OneSignal

What it is: A third-party service wrapping FCM and APNs with a web dashboard for non-developer notification campaigns, A/B testing, and advanced segmentation.

Why not chosen: Introduces a third-party dependency between your server and the user's device. For a project demonstrating the notification pipeline (RabbitMQ → consumer → FCM), using OneSignal would hide the mechanics. FCM is free and has a Go Admin SDK for direct server-side dispatch.

Alternative: Self-Hosted Push (WebSocket-Based)

Why not chosen: Only works when the app is in the foreground (WebSocket connection alive). When the app is closed or in the background, the OS kills the WebSocket to save battery. FCM's value is that Google Play Services maintains a single persistent connection for ALL apps on the device — your notification piggybacks on this always-on connection. Replicating this requires OS-level integration that only Google and Apple have.

Technology 13

bcrypt

Cryptography

What Is bcrypt?

bcrypt is a password hashing function designed by Niels Provos and David Mazières in 1999, based on the Blowfish cipher. Two critical properties: (1) Automatic salt — a unique 128-bit random value generated per hash, so identical passwords produce different hashes. (2) Adjustable cost factor — a parameter (typically 10–12) controlling the number of iterations (2^cost). Higher cost = slower hashing = harder to brute-force.

// A bcrypt hash:
$2b$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
 ↑  ↑  ↑                      ↑
 │  │  │                      └── Hash (31 chars)
 │  │  └── Salt (22 chars, Base64)
 │  └── Cost factor (10 = 2^10 = 1,024 iterations)
 └── Algorithm version (2b)
Usage in Quiz Battle — Current State On registration: bcrypt.GenerateFromPassword(password, bcrypt.DefaultCost). On login: bcrypt.CompareHashAndPassword(storedHash, submittedPassword). On POST /user/change-password: generates a new bcrypt hash from the submitted new password and calls MongoDB UpdateOne to replace password_hash. Google Sign-In users have an empty password_hash — they authenticate via Google ID token, not bcrypt. Dev password recovery: the seed script (cmd/seed/main.go) uses MongoDB upsert semantics — re-running it resets the password_hash for all test users (alice, bob, charlie, diana, evan, fiona, manas1, manas2, manas3) to a fresh bcrypt hash of speakx123. This is the canonical tool for recovering from unknown-password situations without any direct hash writes.

The Rejected Alternatives

Alternative: SHA-256 (or any fast hash)

Speed — the fatal flaw for password storage: SHA-256 is designed to be fast. A modern GPU (NVIDIA RTX 4090) can compute approximately 20 billion SHA-256 hashes per second. An attacker can try every 8-character password (lowercase + digits = 2.8 trillion combinations) in about 2 minutes. No salt by default either — identical passwords produce identical hashes, enabling rainbow table attacks. bcrypt at cost 10 takes ~100ms per hash — 100,000× slower for the attacker, with an auto-generated salt defeating rainbow tables.

When SHA-256 IS the right choice: File integrity verification, digital signatures, HMAC (message authentication), blockchain, content-addressable storage (Git) — any scenario where speed is desired and the input is not a secret.

Alternative: Argon2

What it is: Winner of the Password Hashing Competition (PHC, 2015). Argon2id (the recommended variant) is memory-hard: requires a configurable amount of RAM (e.g., 64 MB) per hash in addition to CPU time. This defeats GPU attacks — GPUs have limited per-core memory, so you can't simply add more cores. bcrypt is CPU-hard but not memory-hard.

Why not chosen: bcrypt is more widely supported in Go's standard ecosystem (golang.org/x/crypto/bcrypt). Argon2 requires an additional dependency. For a project not storing millions of passwords under nation-state-level threats, bcrypt at cost 10–12 is more than sufficient for decades.

When Argon2 IS the right choice: New projects starting from scratch (OWASP recommends Argon2id), applications expecting sophisticated attackers, high-value credential stores.

Alternative: scrypt

What it is: Created by Colin Percival in 2009. Also memory-hard — requires a large pseudo-random array to be kept in RAM during hashing, making GPU attacks expensive. Somewhat superseded by Argon2 as the modern recommendation, but still secure.

Alternative: PBKDF2

What it is: Defined in RFC 2898 (2000). Applies HMAC-SHA256 repeatedly (NIST recommends 600,000+ iterations as of 2023). Not memory-hard — minimal RAM usage makes it vulnerable to GPU and ASIC attacks even at high iteration counts. Still NIST-approved (FIPS 140-2 compliant), making it required in some regulated environments.

Final Verdict — Why bcrypt Deliberate slowness defeats brute-force, automatic salting defeats rainbow tables, wide Go ecosystem support, and paired with the upsert seed script as the canonical dev-recovery tool. For a new production application starting today, Argon2id would be recommended — but bcrypt remains secure and well-understood.

Technology 14 NEW

Client Resilience & Reconnection Architecture

New Flutter gRPC

The Problem

gRPC server-streaming is built on HTTP/2, which is built on TCP. Mobile devices are not on stable connections — they switch between WiFi and mobile data, experience brief packet loss, and have aggressive OS sleep modes that can kill idle TCP connections. When the underlying TCP connection drops, the gRPC stream throws an exception. There is no built-in reconnection in gRPC streaming — reconnect is entirely the client's responsibility.

Without reconnection, one 2-second network blip permanently disconnects a player for the rest of the match. The timer freezes on their screen, no events arrive, and they can only watch a stuck UI. This is a catastrophic user experience for a real-time game.

The Solution: ReconnectService

ReconnectService (flutter-app/lib/services/reconnect_service.dart) wraps any gRPC game-event stream with automatic exponential-backoff reconnection. It is implemented as an async* generator — a Dart function that yields events one at a time, transparently hiding the underlying reconnect attempts from the rest of the application.

// Simplified structure of the resilient stream
Stream<GameEvent> _buildResilientStream(...) async* {
  int attempt = 0;
  while (true) {
    try {
      if (attempt > 0) {
        final leaderboard = await _catchUpState(roomId); // restore scores
        yield ReconnectedEvent(leaderboard: leaderboard);
      }
      final source = _gameService.streamGameEvents(roomId, userId); // new gRPC call
      await for (final event in source) {
        attempt = 0;          // reset on any successful event
        yield event;
        if (event is MatchEndEvent) return; // legitimate end — stop retrying
      }
      return; // clean server close — stop retrying
    } catch (e) {
      attempt++;
      if (attempt > _maxRetries) { yield ReconnectFailedEvent(...); return; }
      yield ReconnectingEvent(attempt, retryInMs: _backoffDelay(attempt));
      yield* _countdownTicks(attempt, delayMs); // UI countdown updates
    }
  }
}

Why Exponential Backoff? Linear backoff (retry every 1 second) would flood the server with reconnect attempts during a network outage. Exponential backoff gives the network time to recover and distributes reconnect load over time. The pattern is used everywhere: TCP retransmission, AWS SDK retries, Google Cloud client libraries.

State Catch-Up on Reconnect

When a player reconnects mid-match, they've missed events that were broadcast to their old channel (which was removed via removeSub when the stream dropped). The _catchUpState method calls getLeaderboard(roomId) before resuming the event stream — restoring current scores without requiring the server to replay past messages.

Server-Side: Late-Joiner Catch-Up

On the server, every time the game loop broadcasts a question event, it also stores it: room.setCurrentQuestion(event, round, deadlineMs). When a player reconnects and calls StreamGameEvents again, addSub registers the new channel. The server can then send getCurrentQuestion() immediately to the new subscriber so they see the current question without waiting for the next round to start.

Question Dedup Guard

When the server sends the current question as a catch-up event to the reconnected player, the Flutter client must not treat it as a new question — that would restart the countdown timer and clear the player's already-submitted answer. The fix is a guard in _onQuestion:

void _onQuestion(QuestionBroadcastEvent e) {
  // Reconnect catch-up guard: server re-sends currentQuestion to late-joiners.
  // If already displaying this exact round+question, skip — don't reset timer
  // or clear the user's selected answer.
  if (state.phase == MatchPhase.inRound &&
      state.currentRound == e.roundNumber &&
      state.currentQuestion?.questionId == e.question.questionId) {
    debugPrint('[GameProvider] catch-up duplicate for round ${e.roundNumber}, skipping');
    return;
  }
  // ... normal question processing
}

The Rejected Alternatives

Alternative: No Reconnection (Fail Permanently)

What happens: When the stream drops, the GameNotifier's onError or onDone fires. The current code transitions to phase = idle with an error message — effectively kicking the player out of the match. For a 30-minute game session, one brief network blip destroys the entire session. Unacceptable UX for a mobile game.

Alternative: Fixed-Interval Retry (No Backoff)

What it is: Retry every 1 second regardless of failure count. Simple to implement.

Why not chosen: If the server is down or overloaded, constant 1-second retries from all clients simultaneously creates a thundering herd — all clients reconnect at the same time, overwhelming the server. Exponential backoff spreads reconnect attempts over time, reducing server load during recovery.

Alternative: WebSocket with Auto-Reconnect Libraries

What it is: Libraries like socket.io-client for Dart/Flutter handle reconnection automatically, with configurable backoff and state restoration hooks.

Why not chosen: The project uses gRPC — switching to WebSocket would require redesigning the entire communication layer, losing Protobuf type safety and auto-generated code. The reconnect complexity is worth it to keep gRPC's benefits. The ReconnectService is 210 lines and achieves the same result.

Alternative: Bidirectional gRPC Streaming

What it is: Instead of server-streaming (client opens → server pushes), use bidirectional streaming where both sides send. The client could send a heartbeat ping every 5 seconds; the server could detect missed pings and proactively close/reopen.

Why not chosen: The reconnect problem is a client-side concern — the server doesn't need to know about reconnects. Server-streaming already gives the server the ability to push all necessary events. Bidirectional streaming would add complexity (client-side ping loop, server-side ping timeout logic) without solving the fundamental problem — the client still needs to reconnect when the stream drops.

Final Verdict — Why This Architecture The ReconnectService wraps gRPC streaming with exponential backoff, state catch-up (getLeaderboard), UI feedback (banner with countdown), and clean termination on MatchEndEvent. The server's currentQuestion mechanism enables instant catch-up for reconnected players. The client-side dedup guard prevents the catch-up event from disrupting an in-progress answer. The combination makes the game resilient to mobile network instability with zero changes to any of the 5 Go services.
